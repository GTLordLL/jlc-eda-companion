# 嘉立创 EDA 工程文件格式分析

基于两个样本工程的逆向分析：`51单片机最小系统板`（标准版）和 `示例-波型产生与变换`（专业版）。

---

## 一、工程文件格式总览

嘉立创 EDA 目前有两代文件格式：

| 版本 | 主文件后缀 | 容器格式 | 备份/导出格式 |
|---|---|---|---|
| **标准版** | `.eprj2` | SQLite 3 数据库 | `.zip` 包 |
| **专业版** | `.epro2` | Zip 压缩包 | 同主文件 |

---

## 二、标准版 (.eprj2) — SQLite 数据库

### 2.1 核心表结构

| 表名 | 用途 | 关键列 |
|---|---|---|
| `projects` | 工程元数据 | `uuid`, `name`, `content`, `owner_uuid`, `thumb` |
| `schematics` | 原理图列表 | `uuid`, `name`, `sheet_count`, `project_uuid`, `display_name` |
| `devices` | 器件定义（符号+封装+属性） | `uuid`, `title`, `display_title`, `source`, `footprint_type`, `symbol_type` |
| `components` | 元件图形数据 | `uuid`, `title`, `docType`, `dataStr`（base64 编码的图形数据） |
| `attributes` | 所有属性键值对 | `key`, `value`, `device_uuid` |
| `project_structures` | 工程结构树（JSON） | `structure`（含 boards/schematics/sheets/pcbs 的层级关系） |
| `documents` | 文档/图纸 | `uuid`, `docType`, `dataStr`, `schematic_uuid` |
| `resources` | 嵌入资源 | `hash`, `dataStr`, `filename` |

### 2.2 LCSC 编号提取路径

```
attributes (key="Supplier Part", value="C6186")
    └── device_uuid → devices.uuid
                         └── devices.title = "AMS1117-3.3_C6186"
                         └── devices.display_title = "AMS1117-3.3_C6186"
                         └── devices.source → 立创云端器件 ID
```

`attributes` 表中关键 key：

| Key | 示例值 | 说明 |
|---|---|---|
| `Supplier Part` | `C6186` | **LCSC 编号（核心）** |
| `LCSC Part Name` | `3.3V 1A` | 立创物料名称 |
| `Manufacturer` | `AMS` | 制造商 |
| `Manufacturer Part` | `AMS1117-3.3` | 制造商型号 |
| `Supplier Footprint` | `SOT-223` | 封装名称 |
| `JLCPCB Part Class` | `Extended Part` | 基础库/扩展库 |
| `Datasheet` | `https://...` | 规格书 URL |
| `Value` | `3.3V` | 参数值 |
| `Designator` | `U?` | 位号前缀 |
| `Add into BOM` | `yes` | 是否加入 BOM |
| `Convert to PCB` | `yes` | 是否转换到 PCB |

### 2.3 实际数据示例

```sql
-- 查询所有器件的 LCSC 编号
SELECT a.value AS LCSC_Part, d.title, d.display_title
FROM attributes a
JOIN devices d ON a.device_uuid = d.uuid
WHERE a.key = 'Supplier Part';
```

结果（51单片机最小系统板）：
```
C500055    SS-3235S-L1      SS-3235S-L1
C6186      AMS1117-3.3      AMS1117-3.3_C6186
C8707      STC89C52RC-40I    STC89C52RC-40I-LQFP44
C295072    XIHCELNANF-12MHZ  XIHCELNANF-12MHZ
C2894939   HDR-M_2.54_1x16P  HDR-M_2.54_1x16P
...（共30个器件）
```

## 三、标准版备份 Zip 结构

```
backup.zip
├── project.json        # 工程元数据 + devices/symbols/footprints/sheets/boards 汇总
├── meta.json           # 版本信息 {"editorVersion": "2.2.43.2", "appVersion": "v3.1.2"}
├── SHEET/{schematic_uuid}/
│   └── {sheet_id}.esch  # 原理图图纸 (NDJSON)
├── PCB/{pcb_uuid}.epcb  # PCB 数据 (NDJSON)
├── SYMBOL/{uuid}.esym   # 原理图符号库 (NDJSON)
├── FOOTPRINT/{uuid}.efoo # PCB 封装库 (NDJSON)
├── BLOB/                 # 二进制资源
├── FONT/                 # 字体文件
├── POUR/                 # 敷铜数据
├── PANEL/                # 拼版数据
└── INSTANCE/             # 复用模块实例
```

`project.json` 的顶层结构：
```json
{
  "schematics": { "uuid": { "name": "原理图", "sheets": [...] } },
  "pcbs": { "uuid": "PCB名称" },

}
```

## 四、核心数据文件格式 — NDJSON (Newline Delimited JSON)

`.esch`、`.epcb`、`.esym`、`.efoo` 四种文件使用相同格式：**每行一个 JSON 数组**。

### 4.1 原理图文件 (.esch)

```
["DOCTYPE","SCH","1.1"]
["HEAD",{"originX":0,"originY":0,"version":"2","maxId":7810}]
```

#### COMPONENT — 原理图元件实例
```json
["COMPONENT","e270","电阻.1",210,745,0,0,{},0]
//          [0]    [1]     [2]X  [3]Y [4]旋转 [5]镜像 [6]{} [7]flag
//          id     名称     坐标        角度    (未使用)
```

#### ATTR — 属性绑定（核心关联数据）
```json
// 位号
["ATTR","e273","e270","Designator","R1",null,1,200,750,null,"st4",0]
//      [0]   [1]    [2]     [3]          [4]  [5] [6][7]X [8]Y [9]  [10]  [11]
//      id    attrId parentId key          value     X   Y       样式  flag

// 器件 UUID 关联
["ATTR","e283","e270","Device","c5d04b2475df4b38b2f34ff206944576",0,0,null,null,0,"st5",0]

// 网络名
["ATTR","e1834","e1832","NET","GND",0,0,345,235,90,"st4",0]

// 全局网络名
["ATTR","e1794","e1792","Global Net Name","VCC+5V",null,null,365,510,null,"st15",0]
```

#### WIRE — 连线（网络拓扑的关键）
```json
["WIRE","e344",[[190,745,190,735],[165,735,190,735]],"st7",0]
//       [0]   [1]线段数组                           [2]样式 [3]flag
//        id   [[x1,y1,x2,y2], [x1,y1,x2,y2], ...]
```

### 4.2 PCB 文件 (.epcb)

```
["DOCTYPE","PCB","1.8"]
["HEAD",{"editorVersion":"2.2.43.2","importFlag":0}]
["CANVAS",0,0,"mil",5,5,5,5,1,1,2,0,5]
["LAYER",1,"TOP","Top Layer",3,"#ff0000",1,"#7f0000",0.5]
```

#### COMPONENT — PCB 元件实例
```json
["COMPONENT","e27",0,1,511.811,-1141.7323,180,{"Unique ID":"gge4","Reuse Block":"","Group ID":"","Channel ID":"$1e530"},1]
//          [0]   [1]层 [2]面 [3]X    [4]Y         [5]旋转 [6]extra                           [7]锁定
//           id

// extra.Channel ID "$1e530" 与原理图 COMPONENT "e530" 关联
```

#### VIA — 过孔
```json
["VIA","e726",0,"GND","",1925,-685,12.0078,24.0158,0,null,null,0,[]]
//      [0]  [1]类型 [2]网络 [3]名 [4]X  [5]Y  [6]内径  [7]外径
```

#### TRACK — 走线（诊断线宽的关键数据）
```json
// TRACK 条目存在于 .epcb 中，表示 PCB 铜皮走线
// 通过解析可获取：走线宽度、所在层、起止坐标
```

## 五、专业版 (.epro2) — Zip 压缩包

```

│   {
│     "title": "示例工程_快速入门",
│     "cbb_project": 0,
│     "editorVersion": "3.2.149.88089769",
│     "introduction": "",
│     "description": "",
│     "tags": "[]"
│   }
├── IMAGE/            # 工程截图
└── {name}.epru       # 主数据文件（核心）
```

### 5.1 .epru 文件格式

每行格式：`{type-header}||{json-body}|`

- 用 `||` 分隔头部和数据体
- 头部含 `type`（DOCHEAD / LAYER / META / ATTR / NET 等）和 `ticket`（序号）
- 行尾以 `|` 结束

#### DOCHEAD — 文档开始标记
```json
{"type":"DOCHEAD","ticket":90}||{"docType":"FOOTPRINT","client":"9cd6a520bbce54c2","uuid":"ae179b9ce5fb4e50982e13006d0c29a1","updateTime":1715851457000,"version":"1638866189","editVersion":"3.2.149.88089769","user":{"uuid":"...","nickname":"ankesiqi","username":"ankesiqi","avatar":""}}|
```

#### LAYER — 层定义
```json
{"type":"LAYER","ticket":1,"id":"[\"LAYER\",1]"}||{"layerType":"TOP","layerName":"Top Layer","use":true,"show":true,"locked":false,"activeColor":"#ff0000","activateTransparency":1,"inactiveColor":"#7f0000","inactiveTransparency":0.5}|
```

## 六、两种格式对比总结

| 维度 | 标准版 (.eprj2) | 标准版备份 (.zip) | 专业版 (.epro2) |
|---|---|---|---|
| 容器 | SQLite | Zip | Zip |
| 原理图数据 | components.dataStr (base64) | .esch NDJSON | .epru 内嵌 |
| PCB 数据 | components.dataStr (base64) | .epcb NDJSON | .epru 内嵌 |
| LCSC 编号位置 | attributes 表 | project.json → devices | .epru → META.attributes |
| 可程序化读取难度 | ⭐⭐ (需 SQLite) | ⭐ (纯文本) | ⭐ (纯文本) |
| EDA 版本 | 标准版 | 标准版 | 专业版 |

```
原理图 ATTR("NET") → "VCC+5V", "GND" 等网络名
```

## 八、对 Skill 开发的关键启示

### 8.1 立创代料检查器 (LCSC Stock Check) — 实现路径

**输入**：嘉立创工程文件路径（自动识别格式）

**处理流程**：
1. 判断格式类型（`.eprj2` / `.epro2` / `.zip`）
2. 提取所有器件的 `Supplier Part`（LCSC 编号）、`Manufacturer Part`、`Supplier Footprint`、`JLCPCB Part Class`
3. 去重，生成 BOM 清单
4. 逐项查询立创商城库存 API
5. 缺货物料 → 搜索 Pin-to-Pin 兼容替代料（相同封装 + 兼容参数）

**技术要点**：

```python
# 标准版：SQL 查询
SELECT a.value, d.title, d.display_title
FROM attributes a
JOIN devices d ON a.device_uuid = d.uuid
WHERE a.key = 'Supplier Part'

import json, zipfile
with zipfile.ZipFile(path) as z:

for dev in data['devices'].values():
```

### 8.2 原理图审查 (ERC) — 实现路径

1. 解析 `.esch` 中 `COMPONENT` → `ATTR("Device")` → `ATTR("Designator")` → `ATTR("Value")` 映射
2. 解析 `WIRE` 条目，重建网络连接拓扑图
3. 解析 `ATTR("NET")` / `ATTR("Global Net Name")` 获取网络名
4. 对照硬件知识库规则检查

### 8.3 PCB 诊断 — 实现路径

1. 解析 `.epcb` 中的 `TRACK`、`VIA`、`PAD`
2. 通过 `COMPONENT.extra.Channel ID` 与原理图关联
3. 检查走线宽度是否满足载流要求、去耦电容是否靠近芯片电源引脚
