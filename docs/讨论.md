## 六、原理图代码能力：可读、可查、可生成

### 原理图本质是代码

嘉立创 EDA 的 `.esch` 文件是 NDJSON（每行一个 JSON 数组），不是图片：

```json
// 元件实例
["COMPONENT","e270","电阻.1",210,745,0,0,{},0]
// 属性绑定
["ATTR","e273","e270","Designator","R1",null,1,200,750,null,"st4",0]
["ATTR","e283","e270","Device","c5d04b2475df...",0,0,null,null,0,"st5",0]
// 连线
["WIRE","e344",[[190,745,190,735],[165,735,190,735]],"st7",0]
```

### 能力矩阵

| 能力 | 可行性 | 说明 |
|------|--------|------|
| **读取**原理图连接关系 | ✅ | 解析 COMPONENT→ATTR→WIRE，重建网络拓扑 |
| **检查**电气规则错误 | ✅ | Phase 2 ERC：对照手册规则验证实际连接 |
| **定位**错误位置 | ✅ | 每个元件/走线都有精确坐标 (x,y) |
| **生成**原理图片段 | ✅ | 输出 NDJSON 行，粘贴到 .esch 文件即可使用 |
| **修改**已有原理图 | ⚠️ | 需要理解嘉立创 EDA 的 undo/redo 和引用完整性 |
| 在嘉立创 EDA GUI 中实时预览 | ❌ | 依赖嘉立创 EDA 插件 API（未知） |
| 自动布局优化 | ❌ | 二维空间问题，纯文本生成容易重叠 |

### 典型场景

**自动添加去耦电容**：
```
用户："给 U1 的每个 VDD 引脚加 100nF 去耦电容"
Claude:
  1. 解析 .esch → 找到 U1 所有 VDD 引脚坐标
  2. 计算电容放置坐标（VDD 引脚旁 5mm）
  3. 生成 NDJSON 代码（COMPONENT + ATTR + WIRE）
  4. 输出 → 用户粘贴到 .esch 文件即可
```

**自动补上拉电阻**：
```
ERC 发现："U1 pin9(NRST) 浮空，手册要求 10KΩ 上拉到 VCC"
Claude:
  1. 定位 pin9 坐标 → 找到最近的 VCC 网络点
  2. 生成 10KΩ 电阻 + 连线的 NDJSON 代码
  3. 输出可粘贴的代码片段
```

## parse_datasheet 重构

全用 docling。

输出架构当前做法（有问题）

```
C8734.pdf (102页)
  → parse_datasheet.py → 单个 JSON
      {
        "full_markdown": "80,000 字符截断...",   ← 大手册被裁掉
        "sections": {
          "pinout": "所有引脚文本混在一起",
          ...
        }
      }
```

问题：
1. 80,000 字符截断 — 大手册（STM32 参考手册可达 1000+ 页）直接丢失
2. 6 个章节塞进一个 JSON — 无法精准定位
3. 章节匹配靠正则猜标题 — 不同厂商格式不同，容易漏
4. 没有目录结构 — 不知道手册有哪些章节

### 目标架构

```
parse_pdf.py → 全文 Markdown (无截断)
    ↓
split_chapters.py → 按章节标题拆分为独立文件
    ↓
datasheets/C8734/
├── index.md                    # 📋 目录索引（完整 TOC）
│   ├── 1. Introduction (p.1)
│   ├── 2. Pin Description (p.5)
│   │   ├── 2.1 Pin Configuration
│   │   └── 2.2 Pin Functions
│   ├── 3. Electrical Characteristics (p.15)
│   └── ...
│
├── 02_pin_description.md       # 独立章节文件
├── 03_electrical_characteristics.md
├── 05_typical_application.md
└── 07_layout_guidelines.md
```

优势：
1. Claude 按需读取 — 需要引脚定义时只读几 KB，不占上下文
2. TOC 索引 — 先读 index.md 了解手册结构，再精准读取
3. 无截断 — 每个文件是完整章节
4. 可追溯 — "依据：手册 §5.2" 可以直接对应到具体文件

### 章节拆分策略

不再像现在这样靠预定义 6 个类别去"匹配"，而是忠实还原手册自身的章节树：

```
步骤1: docling 解析 PDF → 全文 Markdown（保留标题层级）
步骤2: 检测所有标题行 → 构建章节树（层级、页码、标题）
步骤3: 按章节边界切分 → 每个章节写入独立 .md 文件
步骤4: 生成 index.md（完整目录 + 文件映射）
步骤5: 在 index.md 中额外标注 PCB 设计相关的"推荐阅读章节"
```
